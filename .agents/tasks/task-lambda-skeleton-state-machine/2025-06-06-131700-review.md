# SleepAudioProcessor Lambda integrated into Step Functions pipeline

A Python 3.12 Lambda function skeleton is added between the initial DynamoDB write and Polly synthesis in the state machine chain. The function updates the metadata record to `PROCESSING_AUDIO` and returns enriched metadata. CDK wires IAM grants for both the state machine (lambda:InvokeFunction) and the Lambda execution role (DynamoDB read/write). Error handling catches `States.ALL` on the new step, routing to the existing `UpdateStatusFailed` path. Six new tests validate the Lambda resource, environment, IAM, state ordering, and error handling.

**Watch for:** The Lambda handler's event parsing relies on Step Functions passing the raw EventBridge envelope unchanged through two prior states—a fragile coupling that no test validates at the integration level (confirmed). The `processAudioResult` path stores the full LambdaInvoke response (including SDK metadata wrapper) which inflates state payload size toward the 256 KB Step Functions limit on larger pipelines (likely).

## High-level view

The Lambda is inserted into the chain via CDK's `LambdaInvoke` task construct. `ResultPath = "$.processAudioResult"` preserves the original state (including `$.detail`) for downstream Polly and DynamoDB steps, and `AddCatch` wires to the same `UpdateStatusFailed` sink the other steps use.

IAM uses `metadataTable.GrantReadWriteData(processorFunction)` for the Lambda role, giving broader permissions (Scan, Query, DeleteItem, etc.) than the handler actually needs—it only calls `update_item`. This is inherent to CDK's grant helper and acceptable for a skeleton, but should be tightened for production.

The handler assumes the Step Functions input is the raw EventBridge detail structure (`event["detail"]["object"]["key"]`), which is correct given current wiring but creates a contract coupling that will break silently if intermediate steps ever reshape the payload.

Asset path resolution uses a `CallerFilePath`-based computation to produce an absolute path regardless of the working directory during synthesis or testing. This solves the JSII CWD mismatch between `dotnet test` and `npx cdk synth` but couples the asset location to the physical source tree layout.

1. **Event schema coupling** — The Lambda assumes `event["detail"]["object"]["key"]` is populated by raw EventBridge input passing through prior states unchanged. Add an integration test or contract assertion that validates the shape reaching the Lambda, or add defensive extraction with a clear error message naming the expected path. (confirmed)
2. **LambdaInvoke response wrapper inflates state** — CDK's `LambdaInvoke` default (`PayloadResponseOnly = true`) wraps the Lambda response in SDK metadata fields (ExecutedVersion, SdkHttpMetadata, StatusCode). With `ResultPath = "$.processAudioResult"`, this metadata persists in the state for every subsequent step. Consider setting `PayloadResponseOnly = false` with explicit result selection if additional Lambda steps are added. (likely)
3. **DynamoDB grant wider than usage** — `GrantReadWriteData` gives the Lambda Scan, Query, BatchGet, DeleteItem and more, but the handler only calls `update_item`. A narrower custom grant (just `dynamodb:UpdateItem`) would tighten the blast radius. Flag for production hardening. (confirmed)
4. **No unit tests for Lambda handler** — The Python handler has no test coverage. Errors in event parsing (e.g., missing `detail` key) are only caught at runtime. A pytest suite with mocked DynamoDB would cover the critical paths cheaply. (confirmed)

## Event payload threading through the state machine

The state machine receives an EventBridge event shaped as `{ "detail": { "bucket": { "name": "..." }, "object": { "key": "..." } }, ... }`. The `WriteInitialMetadata` step uses `ResultPath = "$.dynamodbResult"`, preserving the original input alongside the DynamoDB response. The `ProcessAudio` step then receives the full state including `$.detail`.

Inside the Lambda, the handler does:

```python
audio_id = event.get("detail", {}).get("object", {}).get("key", "")
bucket_name = event.get("detail", {}).get("bucket", {}).get("name", "")
```

CDK's `LambdaInvoke` defaults to passing the entire state as the Lambda payload. The coupling is implicit: if any upstream step were changed to use `OutputPath` or reshape the state, the Lambda silently receives an empty `audio_id` and raises `ValueError("Missing audioId in event input")`. No test in the suite validates the actual event structure reaching the Lambda—the CDK tests only verify wiring (state names, ARN references, IAM actions), not data flow contracts.

## LambdaInvoke result handling and state size

The `LambdaInvoke` is configured with `ResultPath = "$.processAudioResult"`. Without `PayloadResponseOnly = false`, CDK generates a `Task` state with `Resource: arn:aws:states:::lambda:invoke` and stores the full SDK response (which includes `Payload`, `SdkHttpMetadata`, `SdkResponseMetadata`, `StatusCode`, `ExecutedVersion`) at that path. The downstream steps don't reference `$.processAudioResult` so the extra data is inert now, but it accumulates in the state object for the entire execution lifetime and will compound if additional Lambda steps follow the same pattern.

## Test coverage gaps

Six new CDK assertion tests cover resource existence, IAM grants, state ordering, and error handling. Two existing tests were updated for the new state ordering.

Not tested: the Python handler logic, the actual event payload shape arriving at the Lambda, the `processAudioResult` content structure, or behavior when `TABLE_NAME` is missing at runtime. These are integration/unit test concerns that live outside CDK assertions but represent real failure modes in production.
