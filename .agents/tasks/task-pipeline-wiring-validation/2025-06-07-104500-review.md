# Input validation gate for the sleep audio pipeline

This change introduces a two-layer validation strategy to the Step Functions state machine. A Choice state (`ValidateInput`) sits between the initial DynamoDB write and the Lambda invocation, rejecting files with unsupported extensions before any compute runs. The Lambda itself then performs a second, overlapping validation pass—required-field checks, extension check, and bucket-name verification—before proceeding to update DynamoDB with `PROCESSING_AUDIO` status.

The approach is defense-in-depth: the Choice state is a cheap, zero-compute gate that prevents Lambda invocations on obviously invalid inputs, while the Lambda catches malformed events that the Choice state's simple glob matching cannot detect (missing fields, wrong bucket).

**Watch for:** The Choice state uses case-sensitive `StringMatches` globs (`*.mp3`) while the Lambda performs case-insensitive extension checks (confirmed). A file named `track.MP3` will fail the Choice state and route to FAILED without ever reaching the Lambda. Additionally, the `UpdateStatusFailed` state references `$.error.Cause` in its DynamoDB expression, but when reached via the Choice state's Default path (not via a Catch), no `$.error` field exists in the state input (confirmed)—this will cause a runtime failure.

## High-level view

The state machine gains a Choice state between `WriteInitialMetadata` and `ProcessAudio` that checks the S3 object key against four glob patterns (`.mp3`, `.wav`, `.ogg`, `.txt`). Valid extensions continue to the processing chain; anything else routes directly to `UpdateStatusFailed`. This is the "fast reject" layer that avoids Lambda cold starts for junk uploads.

The Lambda handler was rewritten from a permissive extract-and-proceed pattern to a strict validate-then-proceed pattern. It now fails explicitly on missing fields, unsupported extensions, and unexpected bucket names. The bucket-name check is soft-gated behind the `INPUT_BUCKET_NAME` environment variable—if unset, that check is skipped.

The failure path from the Choice state's Default branch reaches `UpdateStatusFailed`, which attempts to read `$.error.Cause` from the state data. Since no Catch handler placed an `$.error` object into the state when the Choice routes to Default, this DynamoDB UpdateItem will fail at runtime trying to resolve the missing JSONPath.

## Choice state and the missing error context

The `ValidateInput` Choice state routes its Default (invalid extension) path to `UpdateStatusFailed`. That DynamoDB UpdateItem task is defined with:

```csharp
{ ":errorInfo", DynamoAttributeValue.FromString(JsonPath.StringAt("$.error.Cause")) }
```

When execution arrives via a Catch handler, `$.error` is populated by `ResultPath = "$.error"` in the CatchProps. But when execution arrives via the Choice Default path, the state input is the unmodified output of `WriteInitialMetadata`—there is no `$.error` key. Step Functions will throw a `States.Runtime` error when it cannot resolve `$.error.Cause`, which will terminate the execution as an unhandled error (there is no Catch on `UpdateStatusFailed` itself).

The fix is either: (a) insert a Pass state before `UpdateStatusFailed` on the Choice Default path that injects a synthetic error payload (`{ "error": { "Cause": "Unsupported file extension" } }`), or (b) make the Choice Default target a different DynamoDB update task that doesn't reference `$.error.Cause`.

## Case sensitivity gap in the validation gate

Step Functions `StringMatches` is case-sensitive per the ASL specification. The Choice state checks `*.mp3`, `*.wav`, `*.ogg`, `*.txt`. A file uploaded as `podcast.MP3` or `audiobook.Wav` will not match any condition and will be routed to the Default (failure) path. The Lambda, by contrast, calls `.lower()` on the key before checking extensions—so it would accept these files if they ever reached it. The two layers disagree on what constitutes a valid extension.

This matters for real-world S3 uploads where filenames come from user devices with arbitrary casing conventions.

## Extension list coupling

The extension list `SUPPORTED_EXTENSIONS = (".mp3", ".wav", ".ogg", ".txt")` in Python is defined independently from the CDK Choice state conditions in C#. A developer adding `.flac` support would need to update both `CdkBaseStack.cs` (the `Condition.Or` block) and `index.py` (the tuple). The tests don't enforce that these two lists match—they verify specific patterns exist in the synthesized template but have no cross-layer assertion linking them to the Lambda's constant.

## Test coverage gaps

Not tested: the Lambda's Python validation logic has no unit tests. A malformed handler would still pass all 44 CDK tests as long as the file exists at the expected path.

Not tested: the runtime behavior when the Choice Default path reaches `UpdateStatusFailed` with no `$.error.Cause` in the state input. This would only surface during an actual Step Functions execution.
