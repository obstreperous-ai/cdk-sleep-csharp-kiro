# Input validation gate for the sleep audio pipeline (v2)

This change adds a two-layer validation strategy to the Step Functions state machine. A Choice state (`ValidateInput`) sits between the initial DynamoDB write and the Lambda invocation, rejecting files with unsupported extensions before any compute runs. The Lambda performs a second validation passŌĆörequired-field checks, extension check, bucket-name verificationŌĆöbefore proceeding.

The v1 review raised three issues. Two have been fixed: uppercase extension variants (`.MP3`, `.WAV`, `.OGG`, `.TXT`) are now included in the Choice conditions, and a `ValidationFailed` Pass state injects a synthetic `$.error` payload before the Default path reaches `UpdateStatusFailed`. The third issue (duplicate extension lists across C# and Python) was acknowledged as a design trade-off.

**Watch for:** Mixed-case extensions like `.Mp3` or `.wAv` still fail the Choice state since only lowercase and ALL-CAPS variants are matched, while the Lambda accepts any casing (possible). The extension list coupling remainsŌĆöno enforcement mechanism prevents the C# and Python lists from drifting (confirmed, acknowledged).

**Verdict**: APPROVED

## High-level view

The `ValidationFailed` Pass state solves the `$.error.Cause` problem from v1. It injects `{ "Error": "ValidationError", "Cause": "Unsupported file extension" }` into `$.error` before reaching `UpdateStatusFailed`, ensuring the DynamoDB UpdateItem expression resolves correctly regardless of whether execution arrives from a Catch handler or the Choice Default path.

The case-sensitivity fix adds explicit ALL-CAPS patterns to the `Condition.Or` block. This handles the most common real-world case (devices that produce `.MP3` or `.WAV`) but leaves a gap for mixed-case filenames. In practice, S3 keys from most upload paths are either all-lowercase or all-uppercase, making this a low-risk residual gap rather than a blocking concern.

The Lambda's Python validation logic still has no unit tests. A broken handler passes all CDK tests as long as the file exists at the expected pathŌĆöworth noting for follow-up.

## ValidationFailed Pass state and error context injection

The fix introduces a `ValidationFailed` Pass state between the Choice Default branch and `UpdateStatusFailed`:

```csharp
var validationFailedPass = new Pass(this, "ValidationFailed", new PassProps
{
    Result = Result.FromObject(new Dictionary<string, object>
    {
        { "Error", "ValidationError" },
        { "Cause", "Unsupported file extension" }
    }),
    ResultPath = "$.error"
});
validationFailedPass.Next(updateStatusFailed);
```

`ResultPath = "$.error"` places the synthetic object at the same path that Catch handlers use, so both `UpdateStatusFailed` and `PublishFailureNotification` resolve `$.error.Cause` without a `States.Runtime` error.

## Residual case-sensitivity gap

The fix adds `*.MP3`, `*.WAV`, `*.OGG`, `*.TXT` to the `Condition.Or` blockŌĆöeight patterns total. Mixed-case (`*.Mp3`, `*.mP3`) still routes to the failure path. Step Functions' `StringMatches` offers no case-insensitive flag, so covering all permutations would require 2^N patterns per extension (impractical). ASL has no built-in string transformation that could lowercase `$.detail.object.key` before the Choice either. In practice, most S3 upload paths normalize to lowercase at the application layer, making this a theoretical rather than operational risk.
