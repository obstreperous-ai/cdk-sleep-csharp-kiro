# Core audio processing and output wiring

The Lambda handler has been rewritten from a metadata-enrichment stub into a full download-process-upload pipeline. It fetches the input object from S3, routes it through a text-preparation path (for `.txt` files destined for Polly) or a passthrough path (for audio), uploads the result to the output bucket under a `processed/` prefix with a timestamped key, and writes the output location into DynamoDB. On the CDK side, the output bucket is now captured in a variable and the Lambda gains `GrantRead` on the input bucket and `GrantWrite` on the output bucket. Three new CDK tests verify the permissions and env var.

**Watch for:** The Lambda reads entire files into memory (30s timeout, default 128 MB memory) which limits practical file size (confirmed). The audio "processing" path is a pure passthrough with no actual transformation (confirmed). The DynamoDB update writes status `PROCESSING_AUDIO` — the same status the downstream state machine would expect to overwrite with `COMPLETED` — rather than a distinct post-upload status, creating ambiguity about what the record represents at any point in time (confirmed). The CDK permission tests serialize all policies to a single string and assert substrings, meaning they pass if *any* resource in the stack has the action — not necessarily the Lambda role on the correct bucket (confirmed).

## High-level view

The CDK change is minimal and well-targeted: assigning the output bucket to a variable, adding the env var, and calling `GrantRead`/`GrantWrite`. Least-privilege is maintained by scoping permissions to only the buckets the Lambda touches.

The Lambda now performs real I/O — downloading, uploading, and updating DynamoDB in a single try/except — but carries no memory or timeout configuration changes to match the new workload. The 30-second timeout and default memory mean a file over ~100 MB will either OOM or timeout, with no graceful pre-check.

The text-file path produces a JSON document (a Polly synthesis instruction) and writes it to S3 as the "processed output," which means the downstream Polly step still uses its hardcoded placeholder text rather than reading this JSON back. The two paths are disconnected: the Lambda prepares instructions nobody consumes.

The new CDK tests verify that actions like `s3:GetObject` and `s3:PutObject` appear somewhere in the synthesized template's IAM policies but don't assert which principal holds them or which resource ARN they scope to. They'll pass even if a future refactor moves the permission to a different role.

<summary>Issues (5)</summary>

1. **Memory/timeout vs. file size** — Lambda uses default 128 MB memory with a 30-second timeout and loads entire files into memory. Add a memory configuration (512 MB+) or guard against oversized objects before `get_object`.
2. **Polly step ignores Lambda output** — The Polly `CustomState` uses hardcoded placeholder text. The Lambda's text-preparation JSON (written to S3) is never referenced by the Polly step, making the text path a no-op for actual speech synthesis.
3. **Ambiguous DynamoDB status** — The Lambda writes `PROCESSING_AUDIO` after upload completes, which is the same status it wrote before this change when it had only started processing. A distinct `PROCESSED` or `UPLOAD_COMPLETE` status would let operators distinguish "started" from "finished uploading."
4. **Weak permission test assertions** — Tests assert that `s3:GetObject` / `s3:PutObject` appear anywhere in serialized IAM policies. They don't verify the actions are attached to the Lambda role or scoped to the correct bucket ARN.
5. **No partial-failure cleanup** — If DynamoDB update fails after a successful S3 upload, the output object is orphaned with no record pointing to it. No compensating action or dead-letter mechanism exists for this case.

## Lambda memory and timeout constraints

The handler calls `response["Body"].read()` on the full S3 object, holding the entire file in memory. After processing (which for audio is identity), the content lives in memory a second time as `processed_content`. For audio files this means peak memory is roughly 2x the file size. The Lambda is configured with the CDK default of 128 MB and a 30-second timeout. A 70 MB WAV file will OOM; a large file on a cold start will timeout. There is no pre-flight `HeadObject` check to reject oversized inputs before committing to the download.

```python
content = response["Body"].read()  # entire object in memory
processed = content                # second reference for audio path
```

## Disconnect between text path and Polly step

When the input is a `.txt` file, `_process_content` builds a JSON payload containing the text, a voice ID, and output format, then writes that to the output bucket. The intent is clearly to feed Polly, but the state machine's `SynthesizeSpeech` custom state has hardcoded parameters:

```csharp
{ "Text", "Welcome to the sleep audio pipeline. This is a placeholder..." },
{ "VoiceId", "Joanna" }
```

Nothing in the state machine reads `$.processAudioResult` to extract the text. The Lambda's preparation work for `.txt` files produces an artifact that no downstream step consumes.

## DynamoDB status semantics

The `_update_dynamodb_status` function writes `PROCESSING_AUDIO` to the record after the upload to the output bucket has already succeeded. In the prior version of the handler, `PROCESSING_AUDIO` was written at the start of processing. Now it represents "processing complete, output uploaded" — a semantic inversion. The state machine later writes `COMPLETED` via `UpdateStatusCompleted`, but between the Lambda return and that step, the status says `PROCESSING_AUDIO` even though all processing is done. If the Polly or UpdateStatusCompleted steps fail, an operator seeing `PROCESSING_AUDIO` cannot tell whether the output file exists in S3 or not.

## CDK test precision

The three new tests each synthesize the full stack, serialize all IAM policies to a single JSON string, and search for action substrings:

```csharp
var policiesJson = JsonSerializer.Serialize(policies);
Assert.Contains("s3:GetObject", policiesJson);
```

This proves the action exists somewhere in the template, not that it's attached to the Lambda's execution role or scoped to the correct bucket. If a second Lambda or role were added with S3 access, these tests would still pass even if the processor function lost its permissions. The env-var test is more precise (it finds the specific function by logical ID), but the permission tests would benefit from the same specificity.

## No cleanup on partial failure

The handler's try/except block catches any exception after the upload and re-raises it. If `_upload_output` succeeds but `_update_dynamodb_status` raises (throttle, network error), the S3 object exists with no DynamoDB record pointing to it. Over time, these orphaned objects accumulate in the output bucket. The state machine's catch will transition to `UpdateStatusFailed`, which writes `FAILED` to DynamoDB — but without the output key, so the orphaned object remains invisible to any cleanup process.
