# Core audio processing pipeline with size guard and orphan logging (v2)

The Lambda now implements a complete download-process-upload pipeline with a 100 MB `HeadObject` pre-flight guard, 512 MB memory allocation, `PROCESSED` status for semantic clarity between "in-progress" and "upload complete," and structured logging of the output key on DynamoDB failure to support orphan discovery. CDK tests filter policies to those attached to the Lambda execution role and scoped to the correct bucket logical ID. All five v1 issues were addressed (one intentionally skipped as out-of-scope).

**Watch for:** The Polly `CustomState` still uses hardcoded placeholder text disconnected from the Lambda's text-preparation output (confirmed, pre-existing â€” documented but not blocking). The 30-second timeout is unchanged despite the Lambda now performing three sequential network calls (HeadObject + GetObject + PutObject), which could timeout on large files near the 100 MB cap over cold starts (likely).

**Verdict**: APPROVED

## High-level view

The memory bump to 512 MB and the `HeadObject` guard at 100 MB create a workable safety envelope: the Lambda rejects files it cannot process within its resource limits before committing to the download. The 30-second timeout is the remaining constraint â€” a 100 MB file on a cold start with three sequential S3 calls could be tight â€” but reasonable for an initial implementation.

The DynamoDB status progression is now semantically distinct: `PROCESSING` (written by Step Functions at start), `PROCESSED` (Lambda finished uploading), `COMPLETED` (downstream step confirms).

The permission tests locate the IAM policy by verifying it references both the Lambda role's logical ID and the target action, then assert the correct bucket logical ID appears in that same policy document. They'll fail if the permission migrates to a different role â€” a significant improvement over v1's stack-wide string search.

## Timeout arithmetic for large inputs

The Lambda's worst case is a ~100 MB input on a cold start: `HeadObject` (~50ms), `GetObject` streaming 100 MB (1â€“8 seconds depending on Lambda network throughput), `PutObject` uploading 100 MB (same order). Add Python cold start (~500ms), processing (negligible for passthrough), and DynamoDB update (~50ms). On the low end this is ~4 seconds; on the high end with S3 throttling or a slow network path, it approaches 15â€“20 seconds. The 30-second timeout leaves limited margin for retries or slow paths â€” worth revisiting when production traffic patterns emerge.

## Permission test residual gap

The tests don't parse the policy statement's `Resource` array to confirm the action is scoped specifically to the bucket's ARN. If the bucket logical ID appeared elsewhere in the policy document (e.g., in a `Condition`) but the `Resource` was `*`, the test would still pass. In practice CDK's `GrantRead`/`GrantWrite` always scope to the bucket ARN, making a false positive unlikely â€” but it's a theoretical weakness if the grant mechanism changes. (possible)
