# SNS lifecycle notifications for the sleep audio pipeline

The pipeline state machine gains two encrypted SNS topics (completed and failed) and publishes to them as the terminal step on both success and failure paths. The DynamoDB failure record now captures `errorInfo` from the caught error's `Cause` field. Six new CDK assertion tests cover topic existence, encryption, state presence, IAM permissions, and error-info propagation.

**Watch for:** The failure path references `$.error.Cause` (a string) for DynamoDB but passes `$.error` (an object) to the SNS failure message — if `$.error` resolves to an object, the SNS `Message` parameter will contain a JSON-serialized sub-document, which is functional but produces an inconsistent payload contract between the two notification topics (confirmed). The `PublishSuccessNotification` step has no catch handler, so an SNS delivery failure after the DynamoDB COMPLETED write leaves the pipeline in an inconsistent state where DynamoDB says COMPLETED but no notification was emitted (confirmed). The `UpdateStatusFailed` step itself has no catch handler, meaning a DynamoDB failure during error-handling silently terminates the execution without any notification (confirmed).

## High-level view

The notification integration follows a clean append-only pattern: two `SnsPublish` tasks are wired at the tail of the existing success and failure chains, and `GrantPublish` scopes IAM to the specific topic ARNs.

The error-handling topology has an asymmetry: every step in the main chain catches to `UpdateStatusFailed`, but the error-handling chain itself (`UpdateStatusFailed -> PublishFailureNotification`) is unprotected. If either of those steps fails, the execution terminates without any further signal. A DynamoDB throttle during the catch path would produce neither a DynamoDB status record nor an SNS notification.

The `$.error.Cause` reference for DynamoDB works because Step Functions stores the `Cause` field as a string. However, the SNS failure message passes `$.error` (the full error object), so the notification payload shape differs from the success message (`{audioId, status, timestamp}` vs `{audioId, status, error: {Error, Cause}, timestamp}`). Downstream consumers need to handle this asymmetry.

Test coverage validates infrastructure presence (topic count, encryption, state names, IAM actions, errorInfo keyword) but doesn't verify behavioral wiring — it doesn't assert state ordering or catch-target relationships in the definition.

## Error-handling topology and its gap

The catch strategy applies `States.ALL` with `ResultPath = "$.error"` on three steps: `WriteInitialMetadata`, `SynthesizeSpeech`, and `UpdateStatusCompleted`. All route to `UpdateStatusFailed`, which chains into `PublishFailureNotification` — unless the error-handling chain itself fails.

The synthesized definition confirms `PublishSuccessNotification` has `"End": true` with no `Catch` clause. If the SNS service returns a 5xx or the KMS key is unavailable, the execution fails in a terminal state after the DynamoDB record already says COMPLETED. A completed job would never trigger subscribers.

`UpdateStatusFailed` and `PublishFailureNotification` both lack catches. A conditional write failure on DynamoDB (e.g., the item was deleted between steps) or SNS throttling would silently drop the entire error notification path. For production, the standard pattern is to add a catch on terminal notification steps routing to a `Fail` state with explicit cause, which at least surfaces in execution history and CloudWatch alarms.

## SNS message payload asymmetry

The success notification:
```json
{"audioId.$": "$.detail.object.key", "status": "COMPLETED", "timestamp.$": "$$.State.EnteredTime"}
```

The failure notification:
```json
{"audioId.$": "$.detail.object.key", "status": "FAILED", "error.$": "$.error", "timestamp.$": "$$.State.EnteredTime"}
```

`$.error` resolves to the full Step Functions error object `{"Error": "...", "Cause": "..."}`, so the published message contains a nested object rather than flat fields. The DynamoDB path already extracts `$.error.Cause` as a string; using the same approach for the SNS message would produce a consistent flat schema across both topics and simplify downstream parsing.

## Test coverage gaps

The six tests validate presence of resources, properties, and keywords in the serialized template. What they don't cover: the ordering constraint (success notification must follow completed update), the catch-target relationship (catches route to `UpdateStatusFailed` specifically), or the message payload structure. A refactor that accidentally swaps the chain order or routes catches to the wrong state would pass all tests. Adding a test that deserializes the definition JSON and asserts `States.WriteInitialMetadata.Catch[0].Next == "UpdateStatusFailed"` would catch wiring regressions.
