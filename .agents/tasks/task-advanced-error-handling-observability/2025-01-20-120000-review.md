# Error handling, retry policies, and observability for sleep audio pipeline

The change adds multi-layered resilience and monitoring to the Step Functions pipeline: specific error catches on the ProcessAudio Lambda task (Lambda.ServiceException, Lambda.SdkClientException), exponential backoff retries on four task states, X-Ray active tracing on the Lambda function, two CloudWatch Alarms (execution failures and Lambda errors), a CloudWatch Dashboard, and structured JSON logging in the Python handler. Eight new CDK assertion tests cover the additions, and ARCHITECTURE.md is updated.

Watch for: CloudWatch Alarms have no actions configured — they fire but notify nobody (confirmed). The failure path (UpdateStatusFailed, PublishFailureNotification) has no retry or catch, so a transient DynamoDB/SNS error during failure handling silently drops the execution into a terminal failure without notification (confirmed). The `polly:SynthesizeSpeech` IAM policy uses `Resources = ["*"]` which is broader than necessary (confirmed, though Polly doesn't support resource-level constraints for this action).

## High-level view

The retry strategy is well-structured: transient Lambda errors get retried before escalation, DynamoDB tasks retry on all errors, and the Polly CustomState embeds its own retry block in the JSON definition. The separation between retry (automatic transient recovery) and catch (escalation to failure path) is correctly ordered in CDK — retries fire first, catches trigger only after exhaustion.

The observability additions are functional but incomplete. The alarms are fire-and-forget: no `AlarmAction` is wired, so the `failedTopic` SNS topic that already exists goes unused for alarm routing. Operational visibility exists only if someone is watching the CloudWatch console.

The failure path itself is the weakest link in the resilience story. Every happy-path and processing task now retries and catches errors, but UpdateStatusFailed and PublishFailureNotification have neither. A transient DynamoDB throttle during failure recording or an SNS publish timeout will cause the entire execution to fail at the Step Functions level without any structured notification or status update reaching DynamoDB.

## Retry/catch asymmetry on ProcessAudio

The retry block includes `Lambda.TooManyRequestsException` but the specific catch block does not — only `Lambda.ServiceException` and `Lambda.SdkClientException` appear in the catch. This means throttle errors that exhaust all three retry attempts fall through to the `States.ALL` catch rather than the Lambda-specific catch. This is likely intentional (throttle errors after retry exhaustion are treated as generic failures) but the asymmetry is undocumented and could confuse future maintainers reviewing the catch routing.

## Polly CustomState — uncovered error types

The Polly retry block targets only `States.TaskFailed`:

```csharp
{ "ErrorEquals", new[] { "States.TaskFailed" } }
```

`States.TaskFailed` won't catch `States.Timeout` or `States.Permissions`. If the state machine role loses Polly permissions at runtime, or if Polly exceeds the default Step Functions task timeout (unbounded unless explicitly set via `TimeoutSeconds` in the state JSON), those errors bypass this retry and go directly to the `States.ALL` catch. Consider adding `States.Timeout` to the retry errors, or setting an explicit `TimeoutSeconds` on the Polly state to bound execution time. (likely)

## Failure path resilience gap

```
ProcessAudio (error) -> UpdateStatusFailed -> PublishFailureNotification (End)
```

`UpdateStatusFailed` is a DynamoDB UpdateItem with no retry and no catch. `PublishFailureNotification` is an SNS Publish, also unprotected. If DynamoDB throttles the update during a burst of failures, the execution fails at the framework level. The `ExecutionsFailed` alarm would fire — but since that alarm has no action configured, the failure is only visible in the CloudWatch console.

This creates a worst-case scenario: the pipeline fails, the failure is not recorded in DynamoDB, the failure notification is not sent via SNS, and no alarm notification reaches any operator. The only evidence is the Step Functions execution history.

## Alarm action gap

Both alarms are created without `AlarmActions`:

```csharp
new Alarm(this, "StateMachineExecutionFailedAlarm", new AlarmProps
{
    Metric = executionFailedMetric,
    Threshold = 1,
    EvaluationPeriods = 1,
    ComparisonOperator = ComparisonOperator.GREATER_THAN_OR_EQUAL_TO_THRESHOLD,
    AlarmDescription = "Alarm when state machine execution fails"
    // No AlarmActions configured
});
```

The `failedTopic` SNS topic already exists. Wiring it as an alarm action closes the notification loop:

```csharp
AlarmActions = new[] { new Amazon.CDK.AWS.CloudWatch.Actions.SnsAction(failedTopic) }
```

This requires adding the `Amazon.CDK.AWS.CloudWatch.Actions` namespace.

## Event payload exposure in structured logging

The first `_log` call passes `event=json.dumps(event)` at INFO level:

```python
_log("info", "Received event", request_id=request_id, status="RECEIVED", event=json.dumps(event))
```

The event contains the full S3 event detail including bucket name and object key. File paths in an audio pipeline could contain user-identifiable information (e.g., user IDs encoded in S3 key paths). Standard practice is to log event payloads at DEBUG level and only structured metadata (audio_id, bucket) at INFO.

## Test coverage gaps

The new tests verify that retry and catch blocks exist in the serialized state machine definition but don't assert on their numeric parameters. A misconfigured `BackoffRate` of 20.0 instead of 2.0, or `MaxAttempts` of 30 instead of 3, would pass all tests. The X-Ray tracing test is notably better — it navigates the JSON property tree structurally and asserts on the exact value `"Active"`.
