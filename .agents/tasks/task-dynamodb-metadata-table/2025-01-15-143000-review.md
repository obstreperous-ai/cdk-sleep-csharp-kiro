# DynamoDB metadata table and status tracking for sleep audio pipeline

This change introduces a DynamoDB table to record the lifecycle of each audio processing job and weaves it into the existing Step Functions state machine as a three-phase flow: write initial metadata (PROCESSING), run Polly synthesis, then mark COMPLETED—or FAILED on error. IAM is handled via `GrantWriteData`, and the table is configured with on-demand billing, AWS-managed encryption, and point-in-time recovery. Eight new CDK Assertions tests cover the table properties and state machine integration.

**Watch for:**
- The `UpdateStatusFailed` handler references `$.detail.object.key` from the original input, but the catch handler injects error information at `$.error`—after a DynamoPutItem the state output is the DynamoDB response stored at `$.dynamodbResult`, meaning `$.detail.object.key` may no longer be resolvable unless you rely on ResultPath preserving the original input. Needs confirmation of how CDK's `ResultPath` interacts with the catch path. (likely)
- `UpdateStatusCompleted` also reads `$.detail.object.key` but by that point the state output has been mutated by both `$.dynamodbResult` (from PutItem) and `$.pollyResult` (from the Polly task). The `ResultPath` settings should preserve the original input at `$.detail`, but this is fragile—any future task that uses `OutputPath` or omits `ResultPath` will break the chain. (possible)
- No catch handler on `UpdateStatusCompleted`—if the final DynamoDB update fails, the execution fails with no status record of the failure. (confirmed)

## High-level view

The DynamoDB table is keyed on the S3 object key with no sort key, meaning it can only track one processing attempt per object key—re-uploads overwrite the previous record via unconditional PutItem.

The state machine integration adds DynamoDB PutItem and UpdateItem steps around the existing Polly task, with per-step catch handlers routing to a FAILED status updater. The error handling covers WriteInitialMetadata and SynthesizeSpeech but not UpdateStatusCompleted, creating an asymmetry where a failure in the final update leaves the record stuck at PROCESSING.

The test suite verifies resource existence and IAM wiring but doesn't assert on the state machine's control flow (catch transitions, state ordering, ResultPath settings). The behavioral contract—"failures always mark FAILED"—is untested at the infrastructure level.

1. **No catch on UpdateStatusCompleted** — If the DynamoDB UpdateItem for COMPLETED fails, the execution errors out with status stuck at PROCESSING. Add a catch handler or accept the gap explicitly.

<details>
<summary>Details</summary>

### State machine error handling asymmetry

Both `writeInitialMetadata` and `pollyTask` have `AddCatch` pointing to `UpdateStatusFailed`, but `updateStatusCompleted` has no catch. The synthesized ASL confirms this: `UpdateStatusCompleted` has `"End": true` with no `Catch` array. If DynamoDB throttles or rejects the update (conditional check failure, validation error), the Step Functions execution transitions to a terminal failure state with no opportunity to mark the record as FAILED. The metadata record remains at PROCESSING indefinitely.

Catching a failure-of-the-failure-handler creates infinite recursion, so this is a reasonable trade-off—but the gap should be surfaced in monitoring. A CloudWatch alarm on execution failures would catch this case downstream.

### Input path dependencies across states

The chain uses `ResultPath` consistently (`$.dynamodbResult`, `$.pollyResult`, `$.updateResult`) and the catch handlers route error payloads to `$.error`, preserving the original input namespace at `$.detail.*`. The contract is implicit—there's no test verifying that `$.detail.object.key` remains accessible after each state transition.

The Polly `CustomState` sets `"ResultPath": "$.pollyResult"` inside its `StateJson` dictionary, mixing L1 (raw ASL JSON) and L2 (CDK DynamoPutItem/DynamoUpdateItem) patterns in the same chain. The `CustomState` doesn't benefit from CDK's type-safe ResultPath handling—if someone later removes that key from the StateJson, the next state's input changes silently and the UpdateStatusCompleted key lookup breaks.

### PutItem idempotency and concurrent executions

`WriteInitialMetadata` uses `DynamoPutItem` with no `ConditionExpression`. If the same S3 object key triggers the pipeline twice (re-upload, EventBridge replay, concurrent executions), the second PutItem overwrites the first record unconditionally. If the first execution is still in PROCESSING, its metadata is silently replaced with a new PROCESSING record, erasing the earlier execution's history.

### Test coverage gaps

The eight new tests verify table existence, key schema, billing mode, encryption, PITR, DynamoDB resource ARN presence in the state machine definition, and IAM policy actions.

Not tested:
- State ordering (WriteInitialMetadata → SynthesizeSpeech → UpdateStatusCompleted)
- Catch transitions (errors route to UpdateStatusFailed)
- ResultPath configuration on each state
- The UpdateStatusFailed state's parameters (status = "FAILED")
- Table name reference in the state machine definition parameters

The tests assert on string containment in the serialized template rather than structural properties of the ASL definition. This is pragmatic given CDK's Fn::Join rendering, but means a refactor that reorders states or removes a catch would not be caught by the test suite.
