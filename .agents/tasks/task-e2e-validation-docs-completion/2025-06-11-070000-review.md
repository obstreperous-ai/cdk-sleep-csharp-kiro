# End-to-end validation tests and documentation polish for Sleep Audio Pipeline

This change adds 56 C# xUnit tests covering complete pipeline flow (happy path, error scenarios, retry behavior, Lambda configuration, IAM permissions, CloudWatch alarm wiring), 19 Python unit tests for the Lambda handler, and comprehensive documentation updates (README rewrite, ARCHITECTURE.md finalization, new SUMMARY.md, AGENT_GUIDELINES.md updates). The xunit.runner.json disables parallel test execution to work around JSII resource exhaustion. The testing approach verifies the synthesized CloudFormation template as a serialized JSON string rather than through structured CDK assertion matchers for state machine internals.

Watch for:
- String-based state machine assertions are brittle — any formatting or ordering change in CDK's JSON serialization breaks tests without indicating a real regression (confirmed).
- Python tests mock at the module-attribute level but the import pattern (`from index import handler` inside each test method) can mask stale module state between tests (likely).
- Template is re-synthesized in every single test method (56 times), which will compound the JSII resource issue the team already documents (confirmed).

## High-level view

The C# test suite validates Step Functions wiring by serializing the entire StateMachine resource to a JSON string and doing substring searches. The trade-off is fragility: these tests are coupled to exact key ordering and escaping in CDK's JSON output, and they re-synthesize the template per test method rather than sharing a fixture.

The Python test suite mocks boto3 clients and environment variables, covering boundary conditions well (wrong bucket, file too large, missing env vars, unsupported extension). The repeated `from index import handler` inside each method body is unusual and creates a latent risk around module-level client globals if test order or import caching changes.

The IAM permission tests verify that required actions exist in policies, but never assert that overly-broad permissions are absent — a regression granting `s3:*` would pass all tests.

## State machine assertion approach and fragility

The `EndToEndValidationTest` class serializes the CloudFormation `AWS::StepFunctions::StateMachine` resource to a string and searches for substrings like `\"WriteInitialMetadata\":{\"Next\":\"ValidateInput\"`. This pattern is fragile in two ways: (1) a CDK library update that changes JSON key ordering or adds whitespace will break tests without any behavioral change to the infrastructure, and (2) the escaped-quote sequences are hard to read and error-prone to maintain.

The tests already demonstrate a better approach for Lambda properties (`LambdaConfig_HasXRayTracingActive` deserializes to `JsonElement` and uses `GetProperty`), but the state machine tests don't follow the same pattern. Deserializing the definition string into a `JsonDocument` and navigating to specific states by key would survive formatting changes.

The `GetStateMachineDefinitionJson` helper serializes the *entire* resource (including Properties, DependsOn, Metadata) rather than extracting just the `DefinitionString`. Substring matches could theoretically collide with metadata or logical ID strings, though the escaped state names make false positives unlikely in practice.

## Template re-synthesis amplifying JSII pressure

Every test method calls `CreateTemplate()`, which instantiates a new CDK `App`, constructs the full `CdkBaseStack`, and runs `Template.FromStack()`. Across 56 tests, this creates 56 separate JSII runtime interactions. The findings in FEAT-001 explicitly acknowledge that running all 129 C# tests simultaneously causes JSII communication errors. xUnit's `IClassFixture<T>` pattern would allow synthesizing the template once per test class and sharing the `Template` instance, since these tests never mutate the template. This would reduce JSII pressure by ~55x for this class alone.

## Python test isolation and module-level globals

The Lambda handler defines `dynamodb` and `s3_client` at module level. Each test method does `from index import handler` inside the method body rather than importing at the top of the file. The patches target `index.s3_client` and `index.dynamodb` on the module object, which works — but if any future test imports these references directly (`from index import s3_client`), the patch won't intercept calls through the local binding. A top-level `import index` with `index.handler(event, context)` calls would make the patching strategy more obvious and less error-prone.

## IAM assertions: presence without absence

The permissions tests find the first policy matching a condition (`policies.First(p => ...)`) and assert specific action strings are present. A regression that added `s3:*` or `iam:*` to the Lambda role would still pass all existing tests. For infrastructure security tests, adding at least one assertion that the total number of actions per policy statement doesn't exceed an expected ceiling would catch permission creep.

## Documentation gap: xunit.runner.json vs troubleshooting

The README troubleshooting section mentions "JSII runtime errors" and suggests running tests by class filter, but doesn't mention that `xunit.runner.json` already disables parallel execution as a mitigation. Readers may try additional workarounds that are already in place, or may re-enable parallelism not realizing it was intentionally disabled.
