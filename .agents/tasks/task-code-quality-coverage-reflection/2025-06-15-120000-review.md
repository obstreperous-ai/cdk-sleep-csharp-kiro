# Shared fixture optimization, CdkBaseStack modularization, and CI coverage

This change tackles three distinct quality-of-life improvements: test performance optimization via xUnit's `IClassFixture<T>` pattern (collapsing ~70 per-test JSII synthesis operations into 1 shared synthesis), extraction of `CdkBaseStack`'s monolithic constructor into named helper methods with XML doc comments, and addition of code coverage reporting to the CI workflow. A Reflections section is appended to `EXPERIMENT.md` and Python type hints are added to the Lambda handler.

**Watch for:** The `GetProcessorFunction` method ignores its parameter and navigates the construct tree by hard-coded child name ŌĆö a fragile indirection that will break silently if the construct ID changes (confirmed). The CI coverage step uses `grep -oP` (Perl regex), which is unavailable on macOS runners and non-GNU environments (confirmed). The shared fixture exposes a mutable `Template` object across all tests ŌĆö safe today but a latent concurrency hazard if serial execution is ever relaxed (likely).

## High-level view

The test fixture refactoring is the most impactful change: `CdkBaseStackFixture` synthesizes once and shares the `Template` and serialized state machine JSON with all 70 tests via constructor injection. Tests that need distinct environments (`[Theory]` with `dev`/`staging`/`prod`) or that call `app.Synth()` still create fresh stacks, keeping the fixture's scope appropriate.

`CdkBaseStack.cs` is restructured from a ~400-line constructor into a constructor that reads like a table of contents with each concern delegated to a private method. The generated CloudFormation is unchanged ŌĆö this is a behavior-preserving readability refactoring.

The CI coverage step appends a Cobertura-derived summary to `$GITHUB_STEP_SUMMARY`. It parses line-rate and branch-rate from the XML using `grep -oP` and pipes through `bc` ŌĆö functional on ubuntu-latest but non-portable and fragile to XML formatting changes.

<summary>Issues (4)</summary>

1. **GetProcessorFunction ignores its parameter** ŌĆö The method accepts `LambdaInvoke processAudioTask` but uses `this.Node.FindChild("SleepAudioProcessorFunction")` instead. Remove the parameter or actually traverse from the task to avoid misleading callers and coupling to a hard-coded construct ID.
2. **CI coverage uses GNU-only grep -oP** ŌĆö `grep -oP` is not available on macOS or Alpine. If the workflow ever runs on a non-GNU runner, the coverage step fails silently (because of `if: always()` and no `set -e`). Use `sed`, `awk`, or a coverage tool like `reportgenerator` instead.
3. **Shared fixture Template mutability** ŌĆö `Template` and the serialized JSON strings are public get-only properties, but `Template` itself is a mutable CDK Assertions object. If a test ever mutates it (e.g., calling a method that modifies internal state), all subsequent tests see corrupted state. Consider documenting or enforcing immutability.
4. **Coverage step silent failure on missing bc or empty results** ŌĆö If `bc` is not installed or no coverage XML is found, the step produces partial output with no error signal. Adding `set -euo pipefail` or a guard would make failures visible.

## GetProcessorFunction: unused parameter and fragile node lookup

```csharp
private Function GetProcessorFunction(LambdaInvoke processAudioTask)
{
    var functionNode = this.Node.FindChild("SleepAudioProcessorFunction");
    return (Function)functionNode;
}
```

The `processAudioTask` parameter exists to suggest the function is extracted from the task, but the implementation ignores it entirely and does a hard-coded `FindChild` lookup. This creates two risks:

1. If the construct ID `"SleepAudioProcessorFunction"` is ever renamed in `CreateProcessorFunction`, this method throws at synthesis time with a cryptic "child not found" error rather than a compile error.
2. Callers believe they're passing a meaningful reference when they're not ŌĆö a maintenance trap.

The straightforward fix is to either return the `Function` from `CreateProcessorFunction` directly (it already creates one internally) and thread it through the constructor, or use `processAudioTask.LambdaFunction` if the CDK API exposes it.

## CI coverage parsing fragility

The coverage display step:

```bash
LINE_RATE=$(grep -oP 'line-rate="\K[^"]+' "$f" | head -1)
```

This relies on:
- `grep -oP` (Perl regex) ŌĆö not available on macOS/BSD grep or Alpine busybox
- The `line-rate` attribute appearing on a single line in the XML (Cobertura format usually does, but minified output would break it)
- `bc` being installed (it is on ubuntu-latest, but not guaranteed on custom runners)

Since the step runs `if: always()`, failures in the coverage parsing don't fail the workflow, but they produce empty or garbled step summaries with no indication anything went wrong. A more robust approach would use `dotnet-reportgenerator` or `xmllint --xpath`.

## Shared fixture and test isolation

`CdkBaseStackFixture` exposes `Template` (a mutable CDK Assertions reference type) alongside two immutable `string` properties. In the current test code, `Template` is only read via `FindResources`, `HasResourceProperties`, and `ResourceCountIs`. The risk is latent: if a future test calls a method that modifies template state, all subsequent tests see corrupted data. xUnit runs tests within a fixture class sequentially (enforced here via `xunit.runner.json`), so the corruption manifests as order-dependent failures rather than race conditions.
