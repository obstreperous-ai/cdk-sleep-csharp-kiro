# Sleep audio pipeline foundational infrastructure via CDK

Two S3 buckets (input and output) with KMS encryption, versioning, and public access blocked, plus an EventBridge rule that fires on Object Created events from the input bucket and targets a stub SQS queue. The implementation follows TDD: 8 xUnit tests verify resource properties via CDK Assertions, and ARCHITECTURE.md is updated with a "Current Implementation Status" section. The stub queue is an intentional placeholder for a future Step Functions state machine.

**Watch for:** Tests don't actually distinguish between input and output buckets (confirmed), and the stub SQS queue has no encryption or dead-letter queue configured (confirmed).

## High-level view

The stack adds four resources: two S3 buckets, one EventBridge rule, and one SQS queue. Both buckets share identical security posture (KMS_MANAGED, versioned, BLOCK_ALL, DESTROY removal). The input bucket enables EventBridge notifications so S3 emits structured events to the default bus. The rule filters on `aws.s3` / `Object Created` with a detail filter on the input bucket name, routing matched events to the stub queue.

Because both buckets have identical configurations, most tests pass if *either* bucket matches — a regression that removes encryption from the input bucket specifically would go undetected as long as the output bucket still has it. The EventBridge rule test checks source and detail-type but not the bucket-name detail filter, leaving the most security-relevant constraint unverified.

The stub SQS queue lacks encryption-at-rest and a dead-letter queue, inconsistent with the KMS posture established on the buckets and called for in the architecture document.

3. **Stub queue has no encryption** (confirmed) — The architecture document mandates KMS encryption for all data stores. The stub queue uses default settings (no SSE). Messages contain S3 event details (bucket name, object key) in plaintext at rest.
4. **No dead-letter queue on stub target** (confirmed) — If the SQS consumer fails or doesn't exist, messages accumulate with default retention (4 days) and silently expire. A DLQ or alarm would surface delivery failures early.

## Test specificity vs. CDK Assertions semantics

The tests use `HasResourceProperties("AWS::S3::Bucket", ...)` which succeeds if *any* resource of that type matches the property subset. With two identically-configured buckets this is inherently ambiguous. `Stack_HasInputBucketWithKmsEncryption` and `Stack_HasInputBucketWithVersioningEnabled` are named as if they target the input bucket, but they'd pass even if only the output bucket had those properties.

CDK Assertions offers `FindResources` which returns a dictionary keyed by logical ID. A more precise pattern:

```csharp
var buckets = template.FindResources("AWS::S3::Bucket");
var inputBucket = buckets.First(b => b.Key.Contains("SleepAudioInputBucket"));
// assert properties on inputBucket.Value["Properties"]
```

Alternatively, asserting on a property unique to the input bucket — `EventBridgeEnabled` synthesizes a `Custom::S3BucketNotifications` resource — would indirectly confirm identity.

## EventBridge rule detail filter coverage

The rule's event pattern includes a `detail.bucket.name` constraint that scopes it to the input bucket:

```csharp
Detail = new Dictionary<string, object>
{
    { "bucket", new Dictionary<string, object>
        {
            { "name", new[] { inputBucket.BucketName } }
        }
    }
}
```

The test asserts only `source` and `detail-type`. The bucket-name filter is the part that prevents the rule from triggering on *every* S3 Object Created event in the account, yet it's the one part left unverified. Adding a detail assertion (even `Match.ObjectLike` to tolerate CDK tokens) would close this gap.

## Stub queue security posture

The `StubProcessingQueue` receives event payloads containing S3 object keys and bucket names. Without `Encryption = QueueEncryption.KMS_MANAGED`, those messages sit at rest unencrypted — inconsistent with the bucket-level encryption posture established elsewhere in the stack.

A one-line addition (`Encryption = QueueEncryption.KMS_MANAGED`) would bring parity. Alternatively, if the stub is truly throwaway, documenting the intentional omission in a code comment would prevent future reviewers from re-flagging it.

## RemovalPolicy.DESTROY and environment awareness

All four resources use `RemovalPolicy.DESTROY`. The architecture document describes a `cdk deploy -c environment=prod` pattern with different retention policies per environment. When that configuration lands, the test suite has no assertion that would catch an accidental DESTROY in production context — worth adding an environment-parameterized test at that point.
