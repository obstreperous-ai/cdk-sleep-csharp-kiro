# TDD bootstrap for CDK sleep audio pipeline

The change adds a test project, CI workflow, and architecture documentation to an existing empty CDK C# stack. The approach is "TDD scaffold first, implementation later": the test project uses CDK Assertions against the current empty stack, includes a skipped placeholder test for future SQS resources, and a GitHub Actions pipeline validates the build-test-synth cycle on every push. Watch for: the CI workflow will fail on `cdk diff` in any environment without AWS credentials (confirmed); the "no resources" test doesn't actually assert emptiness (confirmed); and the `InternalsVisibleTo` path may break if the test assembly name ever diverges from `CdkBase.Tests` (possible).

## High-level view

The two active tests both pass trivially today since the stack is empty—they function as scaffolding rather than meaningful regression gates. The "no resources" test in particular asserts only that `ToJSON()` is not null, which can never fail.

The CI workflow covers restore → build → test → synth → diff in a single job. The `cdk diff` step uses both `--no-fail` and `|| true` to suppress failures, making it silently swallow credential errors, synthesis errors, or any other genuine problem. Without AWS credentials configured in the workflow, that step is inert.

The architecture documentation describes a planned pipeline (S3 → EventBridge → Step Functions → Lambda → DynamoDB) but the skipped TDD placeholder test targets SQS, which doesn't appear in the architecture doc—a mismatch between the documented plan and the test scaffold's direction.

## Test project: assertion that doesn't assert

The `Stack_HasNoResources` test is named to suggest it verifies the stack is empty, but the implementation only calls `template.ToJSON()` and asserts the result is not null. `ToJSON()` returns the full CloudFormation template (which always includes at least a `Resources: {}` key), so this assertion can never fail. To actually verify emptiness, the test would need to inspect the `Resources` dictionary:

```csharp
template.ResourceCountIs("AWS::CloudFormation::WaitConditionHandle", 0);
// or iterate the template JSON's Resources key and assert Count == 0
```

Without this, the test provides zero protection against accidentally committed resources. (confirmed)

## CI workflow: credentials and failure modes

The synthesis step should work without AWS credentials because the stack is environment-agnostic (the `Env` property is commented out in Program.cs). However, `cdk diff` compares against a deployed stack via CloudFormation API calls, which requires valid AWS credentials. Without them it will error, and `|| true` swallows that error silently.

The `--no-fail` flag tells CDK to exit 0 when differences exist (normally it exits 1 to signal drift). The `|| true` on top of that means *any* failure—including synthesis errors, credential errors, or CDK CLI crashes—is also swallowed. This double suppression makes the step entirely non-functional as a gate. If the intent is "show diff if possible, don't block CI", a comment to that effect would help future maintainers understand this isn't a bug. (confirmed)

## InternalsVisibleTo fragility

The diff adds `[assembly: InternalsVisibleTo("CdkBase.Tests")]` to grant the test assembly access to the internal constructor. If the test project is ever renamed or its assembly name diverges from `CdkBase.Tests`, tests will fail to compile with a non-obvious error about constructor accessibility. (possible)

## Architecture documentation vs. implementation direction

The architecture doc describes S3 → EventBridge → Step Functions → Lambda → DynamoDB. The skipped TDD placeholder test asserts `ResourceCountIs("AWS::SQS::Queue", 1)`. SQS doesn't appear in the architecture. This creates confusion about what the team should implement first when they un-skip the test. Aligning the placeholder with the architecture's first resource (likely an S3 bucket with event notifications) would provide clearer direction. (confirmed)
